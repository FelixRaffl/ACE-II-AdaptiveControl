# Simulink Models for the Adaptive DC Motor Controller

## Purpose

The Stage-1 to Stage-3 models implement the validated Simulink side of the adaptive DC motor controller with RLS estimation, pole placement, and incremental PI control at `T = 0.08 s`. They are generated by [`matlab/build_stage_models.m`](../matlab/build_stage_models.m) and validated headlessly by [`matlab/validate_stage_models.m`](../matlab/validate_stage_models.m). The corresponding validation plot is [`img/simulation_stage3.png`](../img/simulation_stage3.png). This directory also contains the second-order hardware experiment used for the report measurement figures.

## Model Overview

| Model | Stage | Purpose |
|---|---:|---|
| `pwm_test.slx` | 1 (hardware) | Open-loop PWM and direction check on ESP32 I/O |
| `encoder_test.slx` | 2 (hardware) | Encoder-to-speed verification with a PWM spin-up path |
| `adaptive_dcmotor_sim.slx` | 3 (simulation) | Same controller against a simulated S->L->S inertia-change plant |
| `adaptive_dcmotor.slx` | 3 (hardware) | Full closed loop on ESP32 I/O for Monitor and Tune |
| `adaptive_dcmotor_2nd_hw.slx` | 2nd (hardware) | Second-order controller as run on the rig at `T = 80 ms` |

| Category | Models | Execution |
|---|---|---|
| Simulation | `adaptive_dcmotor_sim.slx` | Normal Simulink run, no hardware required |
| ESP32 hardware | `pwm_test.slx`, `encoder_test.slx`, `adaptive_dcmotor.slx`, `adaptive_dcmotor_2nd_hw.slx` | Monitor and Tune with the board connected |

The controller core is the same `Adaptive controller` subsystem in the generated Stage-3 models. The simulation therefore validates the controller code used by the hardware model; only the plant boundary changes from simulated dynamics to ESP32 I/O.

The controller subsystem has two inputs (`reference`, `speed`) and five outputs (`u_sat`, `a0_est`, `b0_est`, `error`, `traceP`). The monitoring outputs feed External Mode scopes and the quantitative validation asserts.

## Prerequisites

- MATLAB R2025b
- Simulink Support Package for Arduino Hardware 25.2.9
- Board: ESP32-WROOM (Arduino Compatible)
- External Mode: XCP on Serial at 921600 baud
- Solver: fixed-step discrete, `T = 0.08 s`, `StopTime = inf`

## First Start and COM Port

The models contain a staged COM-port value from the donor configuration. Before the first hardware run, select the actual ESP32 port in **Hardware Settings -> Target hardware resources**. This setting is local to each lab PC.

## Workflow: `encoder_test.slx`

1. Start **Monitor and Tune** from the Hardware tab and wait for the connection.
2. Turn the shaft by hand. The `Raw count N` display must count up or down depending on direction.
3. Rotate the shaft exactly one revolution. The measured counter difference is 44 counts, matching 11 PPR with x4 quadrature decoding.
4. The gain block `Counts to rad_s` in `encoder_test.slx` and `adaptive_dcmotor.slx` uses `2*pi/(0.08*44)`.
5. For spin-up, keep the Constant `duty` at 0 initially and increase it gradually in the 0..255 range. IN1 and IN2 are fixed to forward direction. The `Bring-up scope` shows raw count `N`, `DeltaN`, and `omega` on stacked axes.

What must hold: `N` is monotonic for a fixed rotation direction, `DeltaN` keeps one sign, and `omega` is plausible. A gearbox-free speed range of 3000-8000 rpm corresponds to about 314-838 rad/s.

## Workflow: `adaptive_dcmotor.slx`

1. Complete the safety checklist in the root [`README.md`](../README.md) before applying motor power.
2. Run the controller with the bare motor. The physical flywheels are not mounted for the submission.
3. Start **Monitor and Tune**. The tunable Constant `omega_ref` starts at 0; increase the reference in small steps.
4. Use the stacked scopes for signals with different scales. The hardware `Adaptation scope` shows `u`, `a0_est`/`b0_est`, `e`, and `traceP`. The `Tracking scope` shows `omega_ref` and `omega`. The `Bring-up scope` shows raw count `N`, `DeltaN`, `omega`, and `IN1`.
5. With sufficient excitation, RLS converges in less than 1 s. `traceP` remains bounded because the excitation gate holds the covariance during idle periods. Per reference step, the design target is around 0.25-0.4 s settling in the validated simulation.

For the initial unit step, inspect the measurement chain on the `Bring-up scope` before closing the loop at higher references. A forward-only check can be run by tuning the saturation lower limit to `0` in External Mode, then restoring `-255` after a stable step.

The physical test bench runs without flywheels for the submission. The complete S->L->S demonstration is validated in `adaptive_dcmotor_sim.slx`: `a0_est` and `b0_est` re-adapt at `alpha = 0.98`, and the critical L->S transition does not create a limit cycle. Hardware settling times are not claimed.

## Tunable Parameters

The parameters are Constants in the `Adaptive controller` subsystem or at top level and are live-tunable in External Mode.

| Parameter | Block | Default | Meaning |
|---|---|---:|---|
| `omega_ref` | Top-level Constant | 0 | Speed reference [rad/s] |
| `alpha` | `Adaptive controller` / `alpha` | 0.98 | RLS forgetting factor |
| `q0`, `q1` | `Adaptive controller` / `q0`, `q1` | 0.06, -0.5 | Desired polynomial `q(z) = z^2 + q1*z + q0` |
| `b0 guard` | `Adaptive controller` / `b0 guard` | 1e-3 | Gain-hold threshold for `abs(b0)` |
| `Counts to rad_s` | Top-level Gain | `2*pi/(0.08*44)` | Encoder speed conversion with measured CPR |
| `duty` | Top-level Constant in `encoder_test.slx` | 0 | PWM duty for Stage-2 spin-up |

`alpha = 1.0` is not used for the hardware demonstration. The selected values are `alpha = 0.98`, `q0 = 0.06`, and `q1 = -0.5`.

## Troubleshooting

- **No connection:** check the COM port, USB cable, and 921600 baud setting. Monitor and Tune performs build and flashing automatically; wait for the full process.
- **`N` does not count:** check GPIO 32/33 wiring, encoder 3.3 V supply, and encoder ground.
- **`omega` has the wrong sign:** swap A/B or the motor leads deliberately and keep the pin assignment documented.
- **Motor does not rotate although `u` is non-zero:** check the ENA jumper, current limit, and static friction threshold.
- **`u` chatters permanently between +/-255:** restore `alpha = 0.98` and verify the measurement chain before raising the reference.
- **Large alternating `omega` spikes:** keep the loop open and inspect encoder sign, `DeltaN`, CPR conversion, and friction-related low-speed cycling before running closed loop.
